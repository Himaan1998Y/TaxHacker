## CRITICAL (1)

### C1 — e-Invoice QR is NOT IRP-compliant
**File**: `lib/e-invoice.ts:34-43`
**Issue**: The QR code uses a pipe-separated 8-field string format. The actual IRP (Invoice Registration Portal) e-invoice QR spec requires the QR to encode the **signed IRN JSON** or a **SHA-256 hash** of the signed invoice — not pipe-separated plaintext. The IRP QR payload must be machine-readable for tax officer verification via the NIC VERIFY app.

**Current format (non-compliant):**
```
27AAOPA1234A1Z5|Business Name|INV-001|01/04/2025|50000.00|1|9983|
```

**Required format (IRP spec):**
The QR should encode a Base64/SHA256 of the signed invoice JSON from IRP, with fields including: `SignedQRCode` from NIC/IRP API response.

**Impact**: Invoices generated by TaxHacker cannot be verified by GST officers via the VERIFY app. This is a legal risk for users — e-invoice non-compliance carries penalties.

**Fix**: Either:
1. Integrate with NIC IRP API to get a real IRN + signed QR, OR
2. Clearly label the QR as "internal reference only" and not "e-Invoice QR" in the UI

## HIGH (4)

### H1 — GSTR-1 missing 5 mandatory sections
**File**: `lib/gstr1.ts:484-564`
**Issue**: The `generateGSTR1JSON` function generates B2B, B2CL, B2CS, HSN, and Nil sections. Missing mandatory sections in GSTN GSTR-1 format:
- **CDNR** — Credit/Debit Notes to Registered (returns, amendments)
- **CDNUR** — Credit/Debit Notes to Unregistered
- **AT** — Advance Tax received (deposits before invoice)
- **ATADJ** — Adjustment of advances
- **DE** — Deemed exports (Section 147 of CGST Act)

**Impact**: The JSON export cannot be directly uploaded to the GSTN portal — it will fail schema validation. Users will see "Missing mandatory section" errors.

### H2 — GSTR-3B missing Table 3.1(d) and 3.1(e)
**File**: `lib/gstr3b.ts:106-153`
**Issue**:
- **Table 3.1(d)** — Inward supplies taxable under Reverse Charge Mechanism (RCM) — not computed. Transactions with `reverseCharge: true` should be reported here as outward liabilities.
- **Table 3.1(e)** — Non-GST outward supplies — not computed.
- **Table 4(A)(3)** — ITC from Imports (IGST paid on imports) — not computed.

**Impact**: GSTR-3B output will have ₹0 for RCM liabilities, potentially causing GST notices for taxpayers who have RCM transactions (landlords, import of services, etc.)

### H3 — Taxable value computed by subtraction instead of from source
**Files**: `lib/gstr1.ts:269,292,334,372`, `lib/gstr3b.ts:182`
**Issue**: Throughout the code: `taxableValue = tx.total - tx.cgst - tx.sgst - tx.igst - tx.cess`

This is arithmetically correct only when the total includes taxes. But if the stored total is just the base amount (tax-exclusive), this gives negative taxable value. The `taxableValue > 0 ? taxableValue : tx.total` fallback silently hides this.

Additionally, floating-point subtraction on accumulated values (as done in GSTR-3B's for-loops) causes rounding drift. With 500 transactions, accumulated error can reach ₹0.50+ — enough to fail GSTN validation.

**Fix**: Store and use `taxableValue` as a dedicated field on Transaction. Don't back-calculate from totals.

### H4 — No Indian Financial Year (FY) enforcement in reports
**File**: `app/api/agent/gstr1/route.ts`, `app/(app)/apps/gstr1/page.tsx`
**Issue**: GSTR-1 and GSTR-3B accept any date range from query params. No validation that the range is within a single Indian FY (April 1 — March 31) or a single GST return period (monthly/quarterly).

A user generating GSTR-1 for "Jan 1 to Dec 31" would get wrong output — mixing two different FYs violates GST return filing rules.

**Fix**: Add FY-aware date range picker with GSTN period validation (MM/YYYY format).

## MEDIUM (5)

### M1 — Nil section format incorrect in GSTR-1 JSON
**File**: `lib/gstr1.ts:556-562`
**Issue**: The nil section is generated as:
```json
"nil": { "inv": [{ "sply_ty": "INTRB2B", "nil_amt": X, ... }] }
```
GSTN format requires separate entries for: `INTRB2B` (intra-state B2B), `INTRB2C` (intra-state B2C), `INTRAB2B` (inter-state B2B), `INTRAB2C` (inter-state B2C). Current code only generates one entry with all nil/exempt aggregated, which is incorrect.

### M2 — HSN description is always empty
**File**: `lib/gstr1.ts:360`
**Issue**: `description: ""` in HSN aggregation. The GSTN portal **requires** HSN description for HSN summary. Empty descriptions cause filing rejection.
**Fix**: Maintain an HSN master lookup table (CBIC publishes HSN master) or at minimum store the description when the transaction is created.

### M3 — Place of supply stored as name, not state code
**File**: `lib/gstr1.ts:572-578`
**Issue**: `placeOfSupply` is stored as a full state name (e.g., "Haryana") but the GSTN JSON requires the 2-digit state code (e.g., "06"). The `getStateCode()` function does a case-insensitive name→code lookup on every transaction, but if the AI extracts "Haryana" vs "HARYANA" vs "haryana", it works — but "Jammu & Kashmir" (GSTN code 01) vs "J&K" would fail and return empty string, silently producing invalid JSON.
**Fix**: Store place of supply as the 2-digit code. Validate against INDIAN_STATES on creation.

### M4 — Floating point accumulation in ITC computation
**File**: `lib/gstr3b.ts:164-184`
**Issue**: All tax amounts are accumulated with `+=` on floating point numbers. With many small transactions, this introduces drift. Example: 100 transactions of ₹18 CGST each should total ₹1800.00 but floating point gives ₹1799.9999...
**Fix**: Use integer (paise) arithmetic throughout; divide only at the end when formatting.

### M5 — GSTIN not validated when saving business settings
**File**: `app/(app)/settings/business/page.tsx` (consumer), `forms/settings.ts`
**Issue**: When a user saves their GSTIN in Settings, the `validateGSTIN()` function exists but it's unclear if it's called at the form validation layer. If a user saves a typo'd GSTIN, all GSTR-1 exports will contain an invalid supplier GSTIN — the most critical field in any GST return.

## LOW (3)

### L1 — Section 194T not in current IT Act
**File**: `lib/indian-tax-utils.ts:206-214`
**Issue**: TDS Section 194T ("Partner Remuneration / Interest") is listed in the TDS_RATES table. As of March 2026, 194T is a proposed section (Finance Bill 2024) but not yet notified by CBDT. Including it may confuse users about its legal status.
**Fix**: Add a note `(proposed, effective from [date])` or conditionally show based on effective date.

### L2 — B2CL threshold hardcoded (not configurable)
**File**: `lib/gstr1.ts:127`
**Issue**: `B2CL_THRESHOLD = 250000` is the current threshold but GSTN occasionally updates this. Hardcoding prevents easy updates when GSTN revises the limit.
**Fix**: Move to config/constants file with a clear comment citing the GSTN circular.

### L3 — No GSTR-2B/GSTR-2A cross-validation
**Issue**: A professional GST filing workflow requires matching inward supplies in GSTR-3B (ITC claimed) against the GSTR-2B data downloaded from the GSTN portal. Without this reconciliation, ITC claims are unverified and prone to mismatches.
**Note**: This is a missing feature, not a bug. But for a "professional CA tool" positioning, this is a notable gap.
