### 2.3 — `gst_rate` Float cast to `::bigint`

**Audit claim**: `lib/stats.ts:499` casts `gst_rate` to `::bigint`, truncating 18.5% to 18.

**Ground truth**: **audit is wrong about which file and which cast**. The actual code at `models/stats.ts:498-501` is:

```sql
COALESCE(gst_rate, 0)::float AS rate,
SUM(COALESCE(cgst, 0))::bigint AS cgst,
SUM(COALESCE(sgst, 0))::bigint AS sgst,
SUM(COALESCE(igst, 0))::bigint AS igst
```

The `::bigint` casts are on the **summed tax amounts** (CGST/SGST/IGST in paise), not on the rate. The rate is cast to `::float`, which is correct. **Summed paise as bigint is the right call** — paise are integers, `SUM()` over ints returns `bigint` by default, and the explicit `::bigint` preserves precision without JS Number truncation at 2^53.

**BUT** — there is a genuine related bug: the comparable query in `models/transactions.ts:183` (the *other* `GSTSummaryResult` — see 2.13) uses `::numeric` for the same COALESCE path and doesn't cast sums at all. So the two GST summary queries use inconsistent precision strategies. Not a truncation bug, but a correctness-consistency bug.

**Status**: 🔶 **Partially real** — the "18.5% truncates" claim is wrong. The real issue is: (a) two GST summary queries with two different precision strategies, (b) `::float` for the rate GROUPS BY float which is technically unstable (floats as GROUP BY keys are a Postgres footgun), (c) merging with 2.13 makes more sense than treating this as standalone.

**Revised scope**: **merge into 2.13** as "GST summary consolidation". One query shape, one type, both call sites use the same helper. Cost inside 2.13.

### 2.4 — Audit log fire-and-forget

**Audit claim**: `lib/audit.ts:46-49` catches errors and only prints to `console.error`. Companies Act 2023 requires immutable audit. A DB blip silently drops audit events.

**Ground truth**: confirmed at `lib/audit.ts:46-49`. Every `logAudit` call site uses the function without `await`-ing a throwable (they rely on the internal catch). 7 call sites total: 6 in `models/transactions.ts`, 1 in `models/settings.ts`.

**Senior reassessment**: the audit's proposed fix (re-throw so the caller fails) is **cure worse than disease**. A transient Postgres blip would block users from creating transactions, which is worse than a dropped audit row. The right shape is:

1. **Best-effort DB write** (current behavior) — don't block user actions.
2. **Local append-only fallback file** when the DB write fails — `/app/data/audit-dlq.jsonl`. The container already mounts `/app/data` as a volume (per Dockerfile), so the file survives restarts.
3. **Drain worker at startup** that reads the DLQ file, replays entries into the DB, and truncates on success.
4. **Startup check** that warns loudly if the DLQ file is non-empty after the drain — indicates persistent DB issues.
5. **Fail-loud path** only when **both** the DB write and the DLQ file write fail — that's a disk-full or permissions issue that needs to surface.

**Out of scope for Tier 2**: a full outbox pattern with periodic re-drain. The DLQ + startup drain is enough for Companies Act — the requirement is retention, not real-time write.

**Status**: ✅ Real but needs senior redesign. This is the **highest-risk item** in Tier 2 because it touches the audit path that runs on every mutation.

**Cost**: 90 minutes including DLQ file code, drain worker, integration tests, and a smoke test that simulates a DB failure mid-write.

### 2.6 — Sentry silent no-op when DSN empty

**Audit claim**: Sentry init runs with empty DSN → silent no-op. No error visibility in prod if DSN is unset.

**Ground truth**: need to untangle this. Three files:
- `next.config.ts:44-55`: `isSentryEnabled = DSN && SENTRY_ORG && SENTRY_PROJECT`. `withSentryConfig` only wraps the Next.js config when all three are present. If any is missing, the config passthrough is the plain `nextConfig`. **This is correct guarding at the build level.**
- `sentry.server.config.ts`: unconditionally calls `Sentry.init({ dsn: process.env.NEXT_PUBLIC_SENTRY_DSN })`. Per Sentry docs (verified via web search), passing an undefined DSN to `Sentry.init` is a **documented no-op** — Sentry disables itself cleanly, no errors thrown, no captures. [Sentry troubleshooting docs](https://docs.sentry.io/platforms/javascript/guides/nextjs/troubleshooting/) confirm this as the recommended pattern for "disable in dev".
- `instrumentation.ts`: dynamic-imports `sentry.server.config` only when `NEXT_RUNTIME === 'nodejs'`. So the init runs but silently no-ops when DSN is empty.

**Senior reassessment**: **the audit is wrong**. Sentry's undefined-DSN behavior is **intended** by the library and **documented** by Sentry as the recommended way to disable in dev. The audit's concern ("no signal that Sentry is broken") is partially valid — but the fix isn't "throw at boot", it's "log a one-line startup message if DSN is empty in production". That converts a silent choice into a visible one without breaking anything.

**Status**: 🟡 **Partially real — downscoped**. The fix is: in `sentry.server.config.ts` and `sentry.edge.config.ts`, log a production-only warning when DSN is empty. Not throw. Do not touch `next.config.ts` — that layer is already correctly guarded.

**Cost**: 15 minutes.

**Cross-cut observation**: same pattern as 0.2 (ENCRYPTION_KEY) and 1.8 (pgvector). Three "silent degradation" items in total. But the **severity gradient is different**: for 0.2 the silent fallback actually stored PII in plaintext (data loss risk), for 1.8 embeddings silently stopped working (feature disabled), for 2.6 error reporting silently disabled (observability loss). The right fix shape depends on the severity. Consolidating them into a single `assertProductionInvariants()` would actually hurt clarity because each has different semantics:
- 0.2: **refuse** to run (data loss)
- 1.8: **graceful degrade** with warning (feature disabled)
- 2.6: **visible acknowledgment** that the optional dependency is off

**Senior call**: **do not consolidate**. Treat these as a pattern in documentation, not a shared helper. The shared helper would conflate three different severity levels.

### 2.9 — Embeddings `$queryRawUnsafe` typed template

**Audit claim**: `lib/embeddings.ts:160-177` uses `$queryRawUnsafe` with string concatenation, returns `as any[]`. Pattern is fragile, one refactor from injection.

**Ground truth**: confirmed. Three sites:
1. `lib/embeddings.ts:121-126` — `storeTransactionEmbedding` — params are positional, **safe today**.
2. `lib/embeddings.ts:159-177` — `findSimilarTransactions` — builds the query string with `+=` for the optional `excludeId` filter. **Safe today** (params are positional) but the concat pattern is one copy-paste away from interpolation.
3. `lib/embeddings.ts:205-219` — `semanticSearch` — clean, single literal template.

**Senior reassessment**: the refactor target is `findSimilarTransactions` — rewrite the `excludeId` branch as a conditional `Prisma.sql` join inside a single `$queryRaw` tagged template. Keeps type safety, removes string concat, zero behavioural change.

**BUT** — Prisma's `$queryRaw` doesn't support `::vector` casts cleanly because pgvector types aren't in Prisma's type system. The current `$queryRawUnsafe` is actually the *pragmatic* choice for pgvector — we can't switch to tagged templates without losing the vector cast. The audit's suggested "migrate to tagged Prisma.sql template" **won't work with pgvector**.

**Status**: 🟡 **Real but unfixable in the proposed way**. The fix is scoped to: remove the `+=` string concat from `findSimilarTransactions` by rewriting the function to two separate SQL literals (one with `excludeId`, one without). Still `$queryRawUnsafe`, still positional params, but no runtime string mutation. Also change the `as any[]` return to a typed row `as Array<{ id: string; ... }>` matching the function's declared return type.

**Cost**: 30 minutes including a test that verifies both branches produce the same result shape.

### 2.10 — Legacy SHA-256 auth cookie cutoff

**Audit claim**: comment says "30-day migration window" but no code-enforced cutoff. Stolen legacy cookies valid forever.

**Ground truth**: confirmed. `lib/self-hosted-auth.ts:48-57` has the `@deprecated` comment. `middleware.ts:136` still accepts legacy tokens via `timingSafeEqual(authCookie, hashSelfHostedToken(password))`.

**Risk analysis**:
- The legacy SHA-256 token is still valid because the migration window was never enforced.
- Our current deployment doesn't track "when was the cookie issued", so we can't distinguish "old cookie from 3 months ago" from "new cookie issued yesterday".
- The safest deprecation path: set a **hard deploy cutoff date**. After 2026-05-01 (arbitrary but concrete), only the new HMAC token is accepted; legacy tokens get rejected and the user re-authenticates with their password.

**User-facing impact**: the single self-hosted admin will be forced to re-login once. Acceptable.

**Senior call**:
- Add a `LEGACY_AUTH_CUTOFF = new Date("2026-05-01")` constant in `lib/self-hosted-auth.ts`.
- In middleware, only accept the legacy token when `Date.now() < LEGACY_AUTH_CUTOFF.getTime()`.
- After the cutoff, the `hashSelfHostedToken` function itself can be deleted — but keep the export for one more release cycle to avoid bundling errors in case any downstream code references it.
- Add a startup log message during the migration window ("legacy cookie tokens still accepted until 2026-05-01").

**Status**: ✅ Real, straightforward with a deploy timing note.

**Cost**: 20 minutes.

### 2.11 — Prisma error leaked to client in agent embeddings

**Audit claim**: `app/api/agent/embeddings/route.ts:84` checks `error.message?.includes("vector")` and returns a detailed error to the client, exposing schema/extension state.

**Ground truth**: confirmed at lines 83-91:

```ts
} catch (error: any) {
  if (error.message?.includes("vector") || error.code === "42883") {
    return NextResponse.json(
      { error: "pgvector extension not enabled. Run the migration first." },
      { status: 501 }
    )
  }
  console.error("Agent API: backfill error:", error)
  return NextResponse.json({ error: "Backfill failed" }, { status: 500 })
}
```

**Two issues here, not one**:
1. Client gets the string `"pgvector extension not enabled"` — which is legitimate info for an authenticated agent, and not sensitive. The audit's concern was hypothetical leakage of schema details, but **this specific string reveals nothing beyond "this feature is off"**, which the agent can already discover via the GET endpoint (lines 103-128 return the same info).
2. The `error: any` + `error.message?.includes` pattern is fragile. A Postgres error with code `42703` (undefined column) would also contain "embedding" but not match this handler and would fall through to the generic 500.

**Interaction with 1.8**: the whole branch is now dead code after Tier 1.8. `storeTransactionEmbedding` no-ops when pgvector is absent, so the backfill loop completes successfully, `processed=0` (all fail text), and nothing throws. The error handler on line 83 only catches errors from the `SELECT` on line 27 — which will throw `42703` (column "embedding" does not exist) in a no-pgvector deployment.

**Senior reassessment**: **this item is substantially obsolete after 1.8**. The remaining fix is:
- Use `isPgvectorAvailable()` from `lib/embeddings.ts` at the top of both GET and POST handlers. If false, return `501 { feature: "disabled" }` without hitting the DB at all.
- Remove the brittle `error.message?.includes("vector")` string-match branch.

**Status**: 🟡 **Real, but obsoleted by 1.8**. Rewrite both handlers to use the explicit capability probe.

**Cost**: 20 minutes.

### 2.13 — Duplicate `GSTSummaryResult` type

**Audit claim**: two different shapes exported under the same name in `models/transactions.ts:164` and `models/stats.ts:468`.

**Ground truth**: confirmed, and the shapes are meaningfully different:

```ts
// models/transactions.ts:164
{
  slabs: Array<{ rate: number; inputGST: number; outputGST: number }>
  totalInput: number
  totalOutput: number
  netPayable: number
}

// models/stats.ts:468
{
  totalInput: number
  totalOutput: number
  netPayable: number
  slabs: Array<{
    rate: number
    inputCGST: number; inputSGST: number; inputIGST: number
    outputCGST: number; outputSGST: number; outputIGST: number
  }>
}
```

**Usage**:
- `components/dashboard/gst-summary-widget.tsx:3` imports from `models/transactions` → the simpler shape.
- Nothing visible imports from `models/stats` for this specific type — but `models/stats.ts:483 getGSTSummary()` returns the detailed shape.

**Senior reassessment**: **two different functions, two different shapes, same name.** This is worse than a typo — it's a semantic collision. The dashboard widget sees a simplified `{rate, inputGST, outputGST}` slab; any code that switches to the `models/stats` version silently gets 6 fields per slab instead of 2, and a TypeScript check would *pass* at the import site but fail at consumption.

**Fix options**:
- **A**: Delete `getGSTSummary` from `models/transactions.ts`, keep the detailed version in `models/stats.ts`, update the widget to consume the detailed shape (computing `inputGST = inputCGST + inputSGST + inputIGST` at the widget).
- **B**: Rename the types: `GSTSummaryTotals` (simple) and `GSTSummarySlabs` (detailed). Keep both functions.
- **C**: Export only from one location, re-export from the other.

**Senior call**: **Option A**. One source of truth for "GST summary". The detailed shape is strictly richer; the simple shape is derivable from it. The widget becomes a pure view layer. Deletes duplicate code.

**Bonus**: also fixes 2.3 (the two inconsistent SQL queries become one query in `models/stats.ts`).

**Cost**: 60 minutes including: (a) consolidate the two SQL queries into one, (b) update the widget, (c) grep for any other callers, (d) update tests if they reference the old shape.
