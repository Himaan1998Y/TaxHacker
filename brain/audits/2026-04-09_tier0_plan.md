## Updated Tier 0 list (7 items, was 5)

Two new items added after researching April 2026 GST rules; one item enhanced.

### 0.1 — B2CL threshold ₹2.5L → ₹1L
- **Why**: Notification 12/2024-CT, effective 1 Aug 2024. Confirmed still current April 2026.
- **Files**: `lib/gstr1.ts:185` (constant), `tests/gstr1.test.ts` (regression test)
- **Change**: One constant + one test case for the ₹1L–₹2.5L band that should now classify as B2CL.
- **Risk**: None — strict tightening; no false positives.

### 0.2 — `ENCRYPTION_KEY` must throw, not warn
- **Why**: Currently boots with PII in plaintext if env var missing. Silent failure class.
- **Files**: `lib/encryption.ts`, `lib/config.ts:51`, ideally a new `lib/assert-invariants.ts` called from `instrumentation.ts`
- **Change**: Throw at startup if `NODE_ENV=production` && key absent or shorter than 32 bytes. Same gate should also handle the other "silent degradation" issues (Tier 2.6 Sentry, Tier 1.8 pgvector) in a single helper.
- **Risk**: Existing prod deployments without the key will fail to boot. **Mitigation**: log clear remediation message; document the env var in `.env.example`.

### 0.3 — Cross-tenant CSV import leak
- **Why**: `findFirst` for category/project omits `userId` filter — user A's import can attach to user B's records.
- **Files**: `models/export_and_import.ts:208-234` (both `importProject` and `importCategory`)
- **Change**: Add `userId` to both `where` clauses. Add a regression test that imports the same category code under two different users and verifies isolation.
- **Risk**: None.

### 0.4 — CI must run `vitest`
- **Why**: 17 test files exist but never run on PRs/main. Every fix can silently re-break.
- **Files**: `.github/workflows/docker-*.yml` (or new `.github/workflows/test.yml`)
- **Change**: New job `test` that runs `pnpm install --frozen-lockfile && pnpm test`. Block docker build job on it. Ideally also `pnpm tsc --noEmit` for type check.
- **Risk**: First run will reveal whatever was already broken. **That's the point** — but we should run locally first before committing the workflow, fix anything red, then enable.

### 0.5 — Remove `"use server"` from `models/files.ts`
- **Why**: Marks every exported function as a publicly-callable Server Action endpoint. Server-side internals exposed as RPC.
- **Files**: `models/files.ts:1` (delete one line)
- **Risk**: If anything in the client tree imports from `models/files.ts` (it shouldn't, but let's grep), it'll break the build. **Mitigation**: grep for client-side imports first; if found, refactor to import via a server action wrapper.

### 0.6 — **NEW** — HSN reporting Phase-III compliance
- **Why**: GSTR-1 Table 12 split into B2B/B2C tabs since April 2025. Our `aggregateHSN()` returns one flat bucket → produces invalid Table 12 data that the portal will reject or misparse.
- **Files**:
  - `lib/gstr1.ts` — `aggregateHSN()` and `generateGSTR1JSON()`
  - `app/(app)/apps/gstr1/components/gstr1-report.tsx` — UI must show two tabs
  - `app/(app)/apps/gstr1/export/route.ts` — CSV export needs `hsn_b2b.csv` + `hsn_b2c.csv`
  - `tests/gstr1.test.ts` and `tests/gstr1-json.test.ts` — regression tests
- **Change**: Split aggregation by classifying each transaction as B2B (has buyer GSTIN → B2B) or B2C (no GSTIN). Two outputs, two tabs.
- **Risk**: Medium — biggest behavioural change of the wave. Test thoroughly before shipping.
- **Source**: [GSTN Phase-III restructuring April 2025](https://rmpsco.com/key-gst-updates-for-april-2025-changes-in-gstr-1-gstr-1a-gstr-3b-you-must-know/)

### 0.7 — **NEW** — GSTR-3B Table 3.2 reconciliation note
- **Why**: GSTN now auto-populates Table 3.2 from GSTR-1; users can't edit it. Our manual computation may contradict the portal.
- **Files**:
  - `lib/gstr3b.ts` — keep computation but flag in output that Table 3.2 is portal-authoritative
  - `app/(app)/apps/gstr3b/components/gstr3b-report.tsx` — banner: "Table 3.2 is auto-populated by GSTN from your GSTR-1 — values shown here are for cross-check only"
- **Change**: UI text + a comment in code. No logic change in v1; we'll align on auto-populated values in a follow-up.
- **Risk**: None — informational only.

## Execution order

The order minimises blast radius and lets each commit ship independently. Each commit gets its own message and PR-equivalent context (we're committing direct to main per the workflow).

1. **0.4 first** (CI gate). Run tests locally. Fix anything red. Push the workflow file. **Now every subsequent commit is gated.**
2. **0.1** (B2CL threshold). Smallest, highest-confidence fix. Write the regression test first; watch CI catch the wrong constant; fix. Validates the gate works.
3. **0.3** (cross-tenant import). Mechanical, well-defined. Add `userId` to two queries + isolation test.
4. **0.5** (remove `"use server"`). One-line delete after grep verification.
5. **0.6** (HSN Phase-III). The big one. Split aggregation, update UI, update CSV export, write tests. May span multiple commits within itself.
6. **0.7** (Table 3.2 banner). Trivial after 0.6.
7. **0.2 last** (ENCRYPTION_KEY throw). Highest deploy risk because it can hard-fail prod boot if env var is missing on the live instance. Save for last so the rest of Tier 0 lands first; verify env is set on Coolify *before* pushing this commit.
